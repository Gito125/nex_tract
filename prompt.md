# Nextract — Phase 9: Desktop Packaging
## Implementation Prompt for LLM

---

## Your Role

You are implementing Phase 9 of Nextract — a local-first media extraction and archiving app.
Nextract is already fully built as a local web app (Phases 1–8 complete). Your job is to wrap it
into a distributable desktop application using Tauri v2.

Read every constraint in this prompt before writing any code. Do not infer, improvise, or add
features beyond what is specified here.

---

## What Nextract Is

A local-first YouTube downloader with:
- Next.js App Router frontend (`web/`)
- FastAPI backend in Python 3.12 managed by `uv` (`server/`)
- SQLite database (`data/nextract.db`)
- yt-dlp called via Python API (not CLI subprocess)
- FFmpeg for audio conversion and format merging
- WebSocket/SSE progress streaming from backend to frontend

The primary flow is: `Paste URL → Analyze → Choose Quality → Download → File saved locally`

---

## Phase 9 Goal

Package Nextract so a user can double-click an icon and the app opens — no terminal, no manual
process starts, no Python or Node.js required on their machine.

**Exit criteria:**
- Nextract runs as a self-contained desktop app on Linux and Windows
- User does not manually start the frontend or backend
- Database is stored in the correct OS user data directory
- Downloads go to the correct OS downloads directory
- App closes cleanly with no zombie processes

---

## Technology Decisions — Do Not Debate These

- **Desktop framework: Tauri v2** (not Electron)
- **Python bundler: PyInstaller** (not Nuitka, not cx_Freeze, not uv standalone scripts)
- **yt-dlp: use the Python API** (`import yt_dlp`) — do not bundle a separate yt-dlp binary
- **FFmpeg: bundle the binary** in `src-tauri/resources/` — do not rely on system FFmpeg
- **Frontend: Next.js static export** (`output: 'export'`) — no Node.js server at runtime
- **Updates: manual only for now** — no Tauri updater plugin in this phase
- **Platforms: Linux and Windows only** — no macOS in this phase

## Task 5 — `src-tauri/src/lib.rs`

This is the core of the desktop packaging. Read it completely before touching it.

```rust
use std::sync::Mutex;
use std::time::Duration;
use tauri::{AppHandle, Manager};
use tauri_plugin_shell::ShellExt;
use tauri_plugin_shell::process::CommandChild;

// Store sidecar child process so we can kill it on exit
pub struct SidecarState(pub Mutex<Option<CommandChild>>);

#[tauri::command]
fn get_api_port() -> u16 {
    // Port the Python backend listens on
    // Must match NEXTRACT_PORT in Python config
    57000
}

pub fn run() {
    tauri::Builder::default()
        .plugin(tauri_plugin_shell::init())
        .plugin(tauri_plugin_dialog::init())
        .plugin(tauri_plugin_fs::init())
        .manage(SidecarState(Mutex::new(None)))
        .setup(|app| {
            let app_handle = app.handle().clone();
            let data_dir = app.path().app_data_dir()
                .expect("Failed to resolve app data directory");
            let resource_dir = app.path().resource_dir()
                .expect("Failed to resolve resource directory");

            // Create data directory if it doesn't exist
            std::fs::create_dir_all(&data_dir)
                .expect("Failed to create app data directory");

            let ffmpeg_name = if cfg!(target_os = "windows") {
                "ffmpeg.exe"
            } else {
                "ffmpeg"
            };

            let ffmpeg_path = resource_dir.join(ffmpeg_name);
            let db_path = data_dir.join("nextract.db");
            let downloads_dir = dirs::download_dir()
                .unwrap_or_else(|| data_dir.join("Downloads"))
                .join("Nextract");

            std::fs::create_dir_all(&downloads_dir)
                .expect("Failed to create downloads directory");

            // Spawn Python sidecar
            let sidecar_cmd = app_handle
                .shell()
                .sidecar("nextract-server")
                .expect("Failed to find nextract-server sidecar")
                .env("NEXTRACT_DATA_DIR", data_dir.to_str().unwrap())
                .env("NEXTRACT_DB_PATH", db_path.to_str().unwrap())
                .env("NEXTRACT_DOWNLOADS_DIR", downloads_dir.to_str().unwrap())
                .env("FFMPEG_PATH", ffmpeg_path.to_str().unwrap())
                .env("NEXTRACT_PORT", "57000")
                .env("NEXTRACT_ENV", "packaged");

            let (_rx, child) = sidecar_cmd.spawn()
                .expect("Failed to spawn Python sidecar");

            // Store child process handle so we can kill on exit
            *app_handle.state::<SidecarState>().0.lock().unwrap() = Some(child);

            // Poll health endpoint until backend is ready, then open window
            let app_handle_clone = app_handle.clone();
            tauri::async_runtime::spawn(async move {
                wait_for_backend_ready(57000).await;
                // Window is already created by Tauri — just show it
                if let Some(window) = app_handle_clone.get_webview_window("main") {
                    window.show().unwrap();
                }
            });

            // Hide window until backend is ready
            if let Some(window) = app.get_webview_window("main") {
                window.hide().unwrap();
            }

            Ok(())
        })
        .on_window_event(|window, event| {
            if let tauri::WindowEvent::CloseRequested { .. } = event {
                // Kill sidecar on window close
                let state = window.state::<SidecarState>();
                let mut guard = state.0.lock().unwrap();
                if let Some(child) = guard.take() {
                    let _ = child.kill();
                }
            }
        })
        .invoke_handler(tauri::generate_handler![get_api_port])
        .run(tauri::generate_context!())
        .expect("Error running Nextract desktop app");
}

async fn wait_for_backend_ready(port: u16) {
    let url = format!("http://127.0.0.1:{}/api/health", port);
    let client = reqwest::Client::new();
    let max_attempts = 30;
    let mut attempts = 0;

    loop {
        attempts += 1;
        if attempts > max_attempts {
            eprintln!("Backend failed to start after {} attempts", max_attempts);
            break;
        }

        match client.get(&url).send().await {
            Ok(resp) if resp.status().is_success() => {
                println!("Backend ready after {} attempts", attempts);
                break;
            }
            _ => {
                tokio::time::sleep(Duration::from_millis(500)).await;
            }
        }
    }
}
```

## Task 13 — Update Frontend API Base URL

The frontend must talk to the Python backend on the correct port.
In packaged mode the port is `57000`. In dev it is `8000`.

Modify `web/lib/api.ts` to resolve the base URL dynamically:

```typescript
// web/lib/api.ts

function getBaseUrl(): string {
  // In Tauri packaged mode, window.__TAURI__ is defined
  if (typeof window !== 'undefined' && (window as any).__TAURI__) {
    return 'http://127.0.0.1:57000'
  }
  // Development fallback
  return process.env.NEXT_PUBLIC_API_URL ?? 'http://localhost:8000'
}

export const API_BASE = getBaseUrl()

export async function apiRequest<T>(
  path: string,
  options?: RequestInit
): Promise<T> {
  const res = await fetch(`${API_BASE}${path}`, {
    headers: { 'Content-Type': 'application/json' },
    ...options,
  })
  if (!res.ok) {
    const error = await res.json().catch(() => ({ detail: 'Unknown error' }))
    throw new Error(error.detail ?? `HTTP ${res.status}`)
  }
  return res.json()
}
```

## Constraints and Rules — Do Not Violate These

**Python / Backend:**
- Never use `shell=True` in any subprocess call
- Never use `pip install` — use `uv add`
- Never expose raw exception strings in API responses
- All file paths must use `pathlib.Path`, never string concatenation with user input
- The config module must remain the single source of truth for all paths and ports
- yt-dlp must be called through `import yt_dlp`, not as a CLI subprocess

**Rust / Tauri:**
- Never hardcode file paths — always resolve via `app.path()` APIs
- The sidecar must be killed cleanly when the window closes — no zombie processes
- The window must remain hidden until the health check passes
- Port 57000 is the packaged port — it must match between Rust and Python config

**Frontend:**
- Never call `fetch` directly in React components — always go through `web/lib/api.ts`
- The `window.__TAURI__` check must be the way packaged vs dev mode is detected
- Do not add `use client` to pages just to access `window` — use a `useEffect` or a
  utility that runs client-side only

**File structure:**
- Do not move any existing files outside the additions described above

## Common Failure Modes to Watch For

**PyInstaller misses yt-dlp plugins:** Add `collect_all('yt_dlp')` to the spec (already in
the spec above). If you still get import errors at runtime, add the failing module path to
`hiddenimports` in the spec.

**Uvicorn workers not found by PyInstaller:** The `hiddenimports` list in the spec covers the
known ones. If you see `ModuleNotFoundError: uvicorn.protocols.http.h11_impl` or similar at
runtime, add the missing module to the spec's `hiddenimports`.

**FFmpeg not found at runtime:** Print `FFMPEG_PATH` in `core/config.py` on startup with a
log statement. Verify the resource path resolves correctly.

**CORS blocking Tauri webview:** Make sure `tauri://localhost` and `https://tauri.localhost`
are both in `CORS_ORIGINS` when `IS_PACKAGED` is true. The exact origin Tauri uses varies
between platforms.

**Window shows blank white before backend ready:** The `window.hide()` / `window.show()` flow
in `lib.rs` handles this. If the window still shows briefly, increase the hide delay or check
that the health poll loop is running before the window is created.

**Port conflict:** Port 57000 is chosen to avoid common dev ports. If something else uses it,
change it consistently in both `lib.rs` (`get_api_port()` and the hardcoded `57000`) and
`core/config.py`.

## Gaps — Linux specific

**1. WebKitGTK version dependency — not mentioned anywhere**

This is the single biggest real-world failure on Linux. Tauri uses the system's WebKit (WebKitGTK). On Ubuntu 22.04 and later it's fine. On Ubuntu 20.04 or any older distro, WebKitGTK may be too old to render your Next.js app correctly — CSS variables fail, certain JS APIs are missing, layout breaks silently.

The prompt says nothing about this. It needs a minimum OS note and a dependency install command.

**2. Linux AppImage requires FUSE — not mentioned**

AppImages need FUSE 2 to mount and run. On Ubuntu 22.04+ FUSE 2 was replaced by FUSE 3, and AppImages fail silently with no error message. The user double-clicks and nothing happens. The fix is either installing `libfuse2` or using the `.deb` target instead. The prompt tells you to test with AppImage but doesn't warn you about this.

**3. `chmod +x` on the PyInstaller output — missing from build script**

The `build-server.sh` copies the binary but doesn't `chmod +x` the entry executable inside the directory. PyInstaller sets permissions correctly on the files it creates, but the `cp` command preserves them — so this is probably fine in practice, but it's not explicitly stated and will silently fail if permissions get stripped somewhere in the pipeline.

**4. Linux build system dependencies — not listed**

To build a Tauri app on Linux you need system packages that are not installed by default on a fresh machine. The prompt has no install command for them. A fresh Ubuntu machine will fail at `pnpm tauri build` with cryptic linker errors.

Missing:
```bash
sudo apt-get install -y \
  libwebkit2gtk-4.1-dev \
  libgtk-3-dev \
  libayatana-appindicator3-dev \
  librsvg2-dev \
  patchelf \
  libssl-dev \
  pkg-config
```

## Gaps — Windows specific

**5. PyInstaller produces false positives with Windows Defender — not mentioned**

This is documented and well-known. PyInstaller's bootloader triggers Windows Defender on many systems — the `.exe` gets quarantined or deleted immediately after build with no explanation. The user tries to run the installer, the `.exe` disappears, the app doesn't start.

The prompt's failure modes section doesn't mention this at all. It needs to tell the implementer:
- This will happen during testing
- The short-term fix is adding the build output directory to Defender exclusions during development
- The production fix is code signing (deferred but at least acknowledged)

**6. Windows build requires Visual Studio Build Tools — not mentioned**

Rust on Windows requires either the full Visual Studio or at minimum the MSVC Build Tools with the "Desktop development with C++" workload. Without it, `cargo build` fails. The prompt tells you to install Rust but says nothing about this prerequisite.

**7. WebView2 runtime on older Windows — partially handled, but wrong**

The prompt's testing checklist says "WebView2 runtime is present (usually is on Win10/11)" but this was removed from the prompt in the final version. More importantly: Windows 10 systems that have never received updates may not have WebView2. Tauri v2 can bundle a WebView2 bootstrapper — the prompt doesn't mention whether to include it.

**8. Windows path separator in `config.py` — subtle bug**

The `config.py` uses `pathlib.Path` correctly everywhere, but there's one place this can break: when Tauri passes the `NEXTRACT_DATA_DIR` env var on Windows, it will contain backslashes (`C:\Users\...`). Python's `pathlib.Path` handles this correctly on Windows but if anywhere in the codebase a path is joined with string concatenation instead of `/` operator, it breaks. The prompt says "use pathlib.Path" but doesn't call this Windows-specific hazard out explicitly.

**9. UPX on Windows — potential false positive amplifier**

The PyInstaller spec has `upx=True`. On Windows, UPX-compressed executables trigger even more antivirus false positives than uncompressed ones. The spec should set `upx=False` for Windows builds. The prompt has one shared spec for both platforms with no conditional.

**10. `build.ps1` is incomplete — not written**

The prompt mentions `scripts/build.ps1` in the target structure and in the build scripts section, but only `build.sh` has actual content. `build.ps1` is never written. An LLM reading this prompt will either skip it or invent something that may not match the `.sh` version.

## Gaps — both platforms

**11. The `--onedir` vs `--onefile` confusion in the build script**

The PyInstaller spec uses `--onedir` mode (a `COLLECT()` call, producing a folder), but the build script copies both the folder AND a single file with the same triple-suffixed name. This is contradictory. Tauri's sidecar mechanism expects either:
- A single executable named `nextract-server-{triple}`, OR
- A folder named `nextract-server-{triple}/` with the executable inside named `nextract-server`

The script tries to do both and gets it half right. A Tauri sidecar with `--onedir` output needs the entire directory bundled, not just the entry executable copied out. This will cause missing `.so`/`.dll` errors at runtime when the sidecar tries to load its dependencies.

**12. Port 57000 is hardcoded in three places with no single source of truth**

It appears in `lib.rs` (the `get_api_port()` function), in `lib.rs` again (the `wait_for_backend_ready` call), and in `config.py`. The prompt says "change it consistently in both places" in the failure modes section, but there are actually three places, not two. The Rust code should define it once as a constant.

**13. Health check timeout — 30 attempts × 500ms = 15 seconds**

If Python takes longer than 15 seconds to start (which can happen on a slow machine with a large PyInstaller bundle), the window never shows. The prompt doesn't note this. The timeout should be longer (60 attempts minimum, giving 30 seconds) or at least documented as a tunable.

**14. No loading screen during backend startup**

The window is hidden until the backend is ready. From the user's perspective: they double-click, nothing happens for 3–10 seconds, then the window appears. No spinner, no "starting up..." message. On a slow machine this looks like the app crashed. The prompt should specify showing a minimal loading window (just the app name and a spinner) during the health poll, then transitioning to the main UI. This isn't mentioned at all.

**15. `API_BASE` is computed at module load time in `api.ts`**

```typescript
export const API_BASE = getBaseUrl()
```

This runs when the module is first imported — during Next.js static export build time. At build time `window` is undefined, so `typeof window !== 'undefined'` is false, and `__TAURI__` is never checked. `API_BASE` will always resolve to the dev fallback, even in the packaged app. It needs to be a function called at request time, not a constant evaluated at import time.
